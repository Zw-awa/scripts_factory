# 离线自动化检查清单

## 离线可行性分级

- 完全离线：所有输入都来自本地文件、本地命令、用户参数或确定性逻辑。
- 部分离线：核心处理在本地，但采集、同步、通知、发布等环节仍依赖网络系统。
- 不适合：核心价值仍依赖实时网页、云 API 或无法替代的托管模型。

如果只能部分离线化，就自动化本地核心部分，并明确剩余在线边界。

## 运行时选择

| 场景 | 优先选择 |
| --- | --- |
| 文件自动化、Windows 设置、计划任务、服务 | PowerShell |
| 结构化文本解析、CSV/JSON 转换、报表生成 | Python |
| 仅依赖标准库的跨平台处理 | Python |
| 对现有本地命令做一层薄封装 | PowerShell 或 shell |

优先选择环境里已经存在、且用户以后最好运行的那一个。

## 需求转脚本清单

1. 列出一次性方案里已经验证过的人工步骤。
2. 把这些步骤收束成一个稳定入口和清晰参数。
3. 把机器相关路径、凭据、阈值移到参数或配置里。
4. 去掉 agent 才能完成的推理部分，改成确定性代码路径。
5. 对可能有破坏性的操作增加安全开关。
6. 为失败场景提供明确的错误输出和退出码。
7. 确保至少存在一个可安全执行的验证入口。
8. 确保脚本自身带有帮助入口，优先内置帮助而不是额外说明文件。

## 建议保留的参数

- `--input`: 主输入文件或目录
- `--output`: 输出文件或目录
- `--config`: 稳定配置项对应的 JSON 配置文件
- `--dry-run`: 只展示计划，不执行真实改动
- `--force`: 覆盖或删除前跳过额外保护
- `--verbose`: 输出更多诊断信息
- `--self-test`: 不依赖真实业务输入的内置安全自检
- `-Help`: PowerShell 脚本的原生帮助开关

只暴露真正对复用有价值的参数。

## 打包默认值

- 优先交付一个可运行脚本，而不是多文件小框架。
- 除非现有流程本来就依赖某个本地库，否则尽量限制在标准库。
- 默认把用法写进脚本帮助，而不是额外再写一堆文档。
- Python 优先使用 `argparse --help`。
- PowerShell 优先提供 `-Help` 和 `Get-Help` 可读的 comment-based help。
- 只有当任务确实存在稳定配置项时，才加入 `config.example.json`。

## 运行验证闭环

1. 运行生成器或成品脚本的 `--help`。
2. 运行 `--self-test`、`-SelfTest`、`--dry-run` 或其他安全验证入口。
3. 若存在小样本且安全，再执行一次接近真实场景的命令。
4. 失败时记录命令、退出码、stdout、stderr。
5. 修改脚本后再次运行同一验证命令。
6. 重复这个闭环，直到通过，或遇到明确外部阻塞。

不要在首次失败后直接停下，更不要把“理论上能运行”当成验证通过。

## 何时承认阻塞

- 本地缺少必须的运行时或依赖。
- 缺少用户必须提供的输入文件、账号信息或配置。
- 沙箱、权限策略或系统策略阻止执行。
- 真正的业务流程本身无法在当前机器上安全复现。

出现阻塞时，说明已经尝试过什么、卡在哪一步、还缺什么。
